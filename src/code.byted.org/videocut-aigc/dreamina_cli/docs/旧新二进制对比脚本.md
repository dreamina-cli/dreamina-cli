# 旧新二进制对比脚本

本文说明如何使用 `scripts/compare_legacy_query_result.sh` 对比旧 `dreamina` 与当前仓库构建产物在 `query_result` 上的输出差异。

## 目标

这个脚本主要解决两个问题：

1. 判断当前程序有没有选错结果媒体。
2. 区分“客户端逻辑不一致”和“服务端重复签发签名 URL”。

在真实环境里，同一个 `submit_id` 连续查询时，服务端可能返回不同域名或不同签名串的 URL，例如：

- `p11-dreamina-sign.byteimg.com`
- `p26-dreamina-sign.byteimg.com`
- `v3-artist.vlabvod.com`
- `v6-artist.vlabvod.com`
- `v9-artist.vlabvod.com`

这类差异不一定代表客户端行为不一致，所以脚本会同时比较：

- 完整 URL
- 归一化后的资源路径 `url.path`
- 主机特定归一化后的 `resource_path`

其中 `resource_path` 会额外处理一类常见情况：

- `vlabvod.com` 视频 URL 的首段 32 位 hash 经常变化
- 同一资源的第二段滚动值也可能变化
- 这段 hash 不一定表示资源变了，很多时候只是签名或分发节点变化

因此脚本会先比较 `resource_path`。如果 `resource_path` 一致，但完整 URL 不一致，会标记为 `signed_url_only`。

## 默认路径

脚本默认使用以下路径：

- 旧二进制：`~/.local/bin/dreamina`
- 新二进制：`dist/<goos>-<goarch>/dreamina`
- 凭证：`~/.dreamina_cli/credential.json`
- 如果不存在，则回退到 `~/.dreamina_cli/b.json`
- 任务库：`~/.dreamina_cli/tasks.db`

执行时会为旧/新二进制分别创建临时 `HOME`，复制：

- `credential.json`
- `tasks.db`
- `tasks.db-shm`
- `tasks.db-wal`

这样可以避免污染当前运行环境。

## 基本用法

单个任务：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id b8cd2dbbcd84b21f
```

多个任务一起跑：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id b8cd2dbbcd84b21f \
  --submit_id d87c6e79531707c3 \
  --submit_id d5277a1572626153 \
  --submit_id dbe58e178e4a4432
```

指定旧/新二进制路径：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --old-bin /Users/saaspit/.local/bin/dreamina \
  --new-bin /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/dist/darwin-amd64/dreamina \
  --submit_id b8cd2dbbcd84b21f
```

指定凭证和任务库：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --credential /Users/saaspit/.dreamina_cli/b.json \
  --tasks-db /Users/saaspit/.dreamina_cli/tasks.db \
  --submit_id dbe58e178e4a4432
```

保留临时目录：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id b8cd2dbbcd84b21f \
  --keep-tmp
```

只输出 JSON 到标准输出：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id b8cd2dbbcd84b21f \
  --json
```

文本输出同时写出 JSON 文件：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id_file /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/testdata/legacy/query_result_submit_ids.txt \
  --json-output /tmp/query_result_compare.json
```

使用固定样本清单批量跑：

```bash
bash /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/scripts/compare_legacy_query_result.sh \
  --submit_id_file /opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/testdata/legacy/query_result_submit_ids.txt
```

默认样本清单位于：

```text
/opt/tiger/src/code.byted.org/videocut-aigc/dreamina_cli/testdata/legacy/query_result_submit_ids.txt
```


`compare` 目前有三种结果：

- `exact_same`
  说明完整 URL 完全一致。
- `signed_url_only`
  说明归一化后的资源路径一致，只差域名、签名串、查询参数或分发层 hash。
- `different_resource`
  说明资源路径本身不同，通常需要继续排查客户端逻辑。

所有样本跑完后，脚本还会输出 summary：


## 推荐排查顺序

当结果不一致时，建议按以下顺序判断：

1. 先看 `status` 和 `queue` 是否一致。
2. 再看 `media_type` 是否一致。
3. 如果 `compare=exact_same`，说明该样本已经完全对齐。
4. 如果 `compare=signed_url_only`，优先判断为服务端重复签名，不要急着改代码。
5. 如果 `compare=different_resource`，再继续检查 `result_json` 合并逻辑、history 解析逻辑和媒体候选优先级。

## 适用场景

这个脚本适合：

- `query_result` 行为回归验证
- 旧程序与新程序的批量样本比对
- 识别签名 URL 抖动
- 发布前做一次高价值 submit_id 回放检查

## 当前限制

- 当前脚本只覆盖 `query_result`
- 当前只比较首个图片或首个视频 URL
- 不直接比较 `credit_count`、完整 `result_json` 和内部恢复字段
- `--json` 模式下标准输出只保留最终 JSON，不再打印逐条文本日志

如果后续需要，可以继续扩展为：

- 比较完整 `result_json`
- 增加 `list_task` 对比
- 增加下载结果的文件名与扩展名对比
